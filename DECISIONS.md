# 아키텍처 및 핵심 기술 의사결정 기록 (DECISIONS.md)

이 문서는 VOW SEOUL 프로젝트의 개발 과정에서 내려진 중요 아키텍처 결정(ADR), 대안, 그리고 그렇게 선택한 기술적 배경과 롤백 조건을 기록합니다.

---

## 1. 루트(`/`) 진입 시 대시보드 리다이렉트

* **결정 사항**: 루트 주소(`/`) 접속 시 소개/홍보용 luxury 랜딩 화면을 배제하고 즉시 관리자 포털 대시보드(`/admin`)로 리다이렉트 처리.
* **결정 이유**:
  * 본 어플리케이션은 일반 대중에게 직접 마케팅하여 가입시키는 B2C 자가 서비스 모델이 아니라, 내부 운영자(어드민) 및 예약 승인된 고객들이 전용 링크를 받아 입력하고 관리하는 B2B/운영 관리 도구의 성격을 띠고 있습니다.
  * 따라서 첫 페이지 접속 시 무거운 애니메이션과 이미지가 포함된 홍보 페이지를 로드하는 것은 비효율적이며, 실제 운영 편의를 위해 즉시 대시보드로 통과시키는 것이 적합하다고 판단했습니다.
* **대안**:
  * 랜딩 페이지를 유지하고 우측 상단에 '로그인' 및 '대시보드 이동' 버튼 제공. (운영 동선에 불필요한 마우스 클릭을 한 단계 더 유발하므로 제외)
* **되돌릴 조건 (Rollback Condition)**:
  * 향후 일반인 예비부부가 웹사이트에 직접 가입하여 템플릿을 선택하고 자가 결제하는 '셀프 모바일 청첩장 패키지' 상품군을 정식 런타임 출시하게 될 경우 홍보용 루트 랜딩 화면으로 복구해야 합니다.

## 2. 별도 주문(Orders) 테이블 없이 고객 및 청첩장 정보 조인을 통한 주문 생성

* **결정 사항**: 데이터베이스에 별도 `orders` 테이블을 생성하지 않고, `customers` 및 `invitations` 테이블을 in-memory 상에서 조인 매핑하여 주문 및 통계 데이터 생성.
* **결정 이유**:
  * 현재 VOW SEOUL의 비즈니스 구조 상 '고객 1명 = 모바일 청첩장 1건 (기본 금액 50,000원)'의 1:1 결제 관계가 성립합니다.
  * 별도 결제 이력 관리용 `orders` 테이블을 구성하면 신규 마이그레이션 및 PG 결제 연동 시 동기화 예외 등의 공수가 크게 증가합니다.
  * 따라서 '폼 제출을 마친 고객' 및 '청첩장이 임시 생성/발행된 고객'의 상태를 추적해 주문과 동등하게 간주하여 매핑하는 방식이 DB의 정규성을 해치지 않으며 기한 내 안정적으로 기능을 공급하는 최선의 방식이었습니다.
* **대안**:
  * Supabase DB에 `orders` 테이블 신설 및 외래키 연동. (단기 설계 리소스 증가 및 복잡도 우려로 연기)
* **되돌릴 조건 (Rollback Condition)**:
  * 장바구니 기능 추가, 옵션별 추가금 부과(예: 사진 보정 서비스 추가, BGM 음원 추가 구매 등), 또는 세금 계산서/영수증 별도 발행 등 주문 단위의 세부 속성이 다각화되는 경우에는 독립적인 `orders` 스키마 구축으로 전환해야 합니다.

## 3. 모바일 입력 폼의 더블 레이어 임시저장 (Local Storage + DB)

* **결정 사항**: 모바일 고객 입력 폼(`form/[slug]/page.tsx`)에서 입력값이 바뀔 때마다 브라우저 `localStorage`에 즉시 자동 저장하고, 저장 버튼 클릭 시에만 Supabase DB에 최종 송신하는 구조 구축.
* **결정 이유**:
  * 모바일 환경(카카오톡 인-앱 웹뷰, Safari 등)에서는 사용자가 은행 앱을 열어 계좌번호를 확인하거나, 예식장 주소를 복사하기 위해 잠시 화면을 나갔다 돌아올 때 페이지 상태가 초기화(킬)되는 현상이 빈번합니다.
  * 매 키 입력마다 Supabase DB로 API 요청을 보내면 데이터베이스 커넥션 풀이 초과될 리스크가 있고 지연 시간이 발생합니다.
  * 따라서 로컬 브라우저 저장소에 백업하여 휘발을 완벽하게 방어하고, 수동 또는 최종 제출 단계에서 DB와 동기화하는 구조가 안전합니다.
* **대안**:
  * Debounce(예: 1~2초 간격)를 활용해 Supabase DB에만 주기적으로 자동 저장. (인터넷 연결이 불완전한 이동 환경에서 유실 가능성 잔존하므로 탈락)
* **되돌릴 조건 (Rollback Condition)**:
  * 기기 교체 입력 시나리오(예: 모바일로 작성하다 데스크톱으로 마저 작성)를 원활하게 대응해야 할 경우에는 서버 측 세션에 임시 상태(Draft Session Table)를 두고 동기화하는 구조로 마이그레이션해야 합니다.

## 4. Supabase DB 단일 행 조회 시 `.maybeSingle()` 사용

* **결정 사항**: 로고 설정 및 기능 활성화 등 설정 테이블 조회 시 기존 `.single()`을 제거하고 `.maybeSingle()`로 전면 변경.
* **결정 이유**:
  * Supabase Client(PostgREST)에서 `.single()`은 반드시 1개의 행이 존재해야만 에러를 내지 않습니다. 0개의 행이 존재하면 HTTP 406 Not Acceptable 에러와 함께 컴포넌트 렌더링 자체가 터지는 현상이 발생했습니다.
  * 초기 설치 혹은 설정값이 채워지지 않은 상태에서도 시스템이 안전하게 작동(Null 값 반환을 통한 기본 프리셋 처리)하려면 무조건 `.maybeSingle()` 처리가 동반되어야 안전합니다.
* **대안**:
  * `.select()` 후 결과 배열의 첫 번째 인덱스 검사 (`data?.[0]`). (코드가 장황해지고 Supabase 라이브러리 인터페이스의 장점을 살리기 위해 `maybeSingle()` 채택)
* **되돌릴 조건 (Rollback Condition)**:
  * 없음 (안정성을 위한 최적의 조치로 고정).

## 5. 어드민 응답 상세 페이지의 복사 모드 (Drag-Select) 기본화

* **결정 사항**: 어드민 제출 응답 확인 화면(`/admin/forms/responses/[instanceId]`)을 기본적으로 텍스트 선택 및 드래그가 가능한 일반 HTML 텍스트 형태로 표시하고, '수정하기' 클릭 시에만 입력 필드(`<input>`)로 전환되도록 설계.
* **결정 이유**:
  * 어드민 작업자의 주요 업무는 고객이 제출한 예식 장소명, 부모 이름, 안내 문구 등을 그대로 긁어서 복사하여 물리 지류 인쇄 파일이나 어도비 일러스트레이터 레이아웃에 붙여넣는 작업입니다.
  * 모든 필드가 일반 입력창(`<input>`, `<textarea>`)으로 감싸져 있으면 특정 단어만 선택(Drag)해서 복사하는 동작이 매우 불편하며 오클릭 시 원치 않게 데이터가 지워지는 문제가 발생합니다.
  * 따라서 평상시에는 텍스트 선택이 최적화된 상태를 유지하고, 의도적으로 수정 모드에 진입했을 때만 편집기가 활성화되게 구성했습니다.
* **대안**:
  * 입력 필드를 유지하고 각 필드 우측에 [복사하기] 단축 버튼 제공. (안내 문구나 축약어 등 일부분만 골라 복사하는 커스텀 드래그 요구를 충족하지 못해 기본 선택 구조 채택)
